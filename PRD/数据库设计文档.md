# 都柏林共享单车与天气服务 - 数据库设计文档

## 数据库概述

数据库名称：`dublin_service`
字符集：`utf8mb4`
排序规则：`utf8mb4_unicode_ci`

## 表结构设计

### 1. bike_station（自行车站点表）

**用途**：存储自行车站点的静态信息，数据不经常变化

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | BIGINT | PRIMARY KEY, AUTO_INCREMENT | 主键ID |
| number | INTEGER | UNIQUE, NOT NULL, INDEX | 站点编号 |
| address | VARCHAR(128) | | 站点地址 |
| banking | INTEGER | | 是否支持银行卡支付（0/1） |
| bike_stands | INTEGER | | 车位总数 |
| name | VARCHAR(128) | | 站点名称 |
| position_lat | FLOAT | | 纬度 |
| position_lng | FLOAT | | 经度 |

**索引**：
- `idx_number`：按站点编号查询（唯一索引）

**数据来源**：JCDecaux API - https://api.jcdecaux.com/vls/v1/stations
**更新频率**：首次初始化后不再更新
**数据保留**：永久保留

---

### 2. bike_station_status（自行车站点状态表）

**用途**：存储自行车站点的动态状态信息（可用车辆数、空余车位数等）

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | BIGINT | PRIMARY KEY, AUTO_INCREMENT | 主键ID |
| number | INTEGER | NOT NULL, INDEX | 站点编号（关联bike_station） |
| last_update | DATETIME | NOT NULL, INDEX | 最后更新时间 |
| available_bikes | INTEGER | | 可用自行车数量 |
| available_bike_stands | INTEGER | | 可用车位数量 |
| status | VARCHAR(128) | | 站点状态（OPEN/CLOSED） |

**索引**：
- `idx_number`：按站点编号查询
- `idx_last_update`：按更新时间查询
- `idx_number_last_update`：复合索引，优化按站点查询最新状态

**数据来源**：JCDecaux API
**更新频率**：每5分钟
**数据保留**：保留7天，自动删除7天前数据

---

### 3. current_weather（实时天气表）

**用途**：存储都柏林的实时天气信息

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | BIGINT | PRIMARY KEY, AUTO_INCREMENT | 主键ID |
| dt | DATETIME | NOT NULL | 天气数据时间（API返回的时间戳） |
| main | VARCHAR(256) | | 天气主要状况（如：Clouds, Rain） |
| description | VARCHAR(256) | | 天气详细描述（如：broken clouds） |
| temp | FLOAT | | 温度（摄氏度） |
| wind_speed | FLOAT | | 风速（m/s） |
| snapshot_time | DATETIME | NOT NULL, INDEX | 数据入库时间 |

**索引**：
- `idx_snapshot_time`：按入库时间查询最新数据

**数据来源**：OpenWeatherMap API - https://api.openweathermap.org/data/2.5/weather
**更新频率**：每1小时
**数据保留**：保留7天，自动删除7天前数据

---

### 4. forecast_weather（预测天气表）

**用途**：存储都柏林未来3天的天气预报

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | BIGINT | PRIMARY KEY, AUTO_INCREMENT | 主键ID |
| future_dt | DATETIME | NOT NULL, INDEX | 预测日期时间 |
| main | VARCHAR(256) | | 天气主要状况 |
| description | VARCHAR(256) | | 天气详细描述 |
| temp_12 | FLOAT | | 12点温度 |
| temp_min | FLOAT | | 最低温度 |
| temp_max | FLOAT | | 最高温度 |
| wind_speed_max | FLOAT | | 最大风速 |
| snapshot_time | DATETIME | NOT NULL, INDEX | 数据入库时间 |

**索引**：
- `idx_snapshot_time`：按入库时间查询
- `idx_future_dt`：按预测日期查询
- `idx_snapshot_future`：复合索引，优化查询最新预报

**数据来源**：OpenWeatherMap API - https://api.openweathermap.org/data/2.5/forecast/daily
**更新频率**：每1小时
**数据保留**：保留7天，自动删除7天前数据

## 数据清理策略

### 自动清理机制

系统通过定时任务（每天凌晨2点）自动清理7天前的数据：

1. **bike_station_status**：删除 `last_update < NOW() - 7天` 的记录
2. **current_weather**：删除 `snapshot_time < NOW() - 7天` 的记录
3. **forecast_weather**：删除 `snapshot_time < NOW() - 7天` 的记录

### 手动清理

可以通过存储过程手动触发清理：

```sql
CALL clean_old_data();
```

## 初始化步骤

1. 执行 `schema.sql` 创建数据库和表结构
2. 执行 `init-data.sql` 创建存储过程
3. 启动应用程序，系统会自动：
   - 首次调用API获取所有站点信息并存入 `bike_station` 表
   - 开始定时任务，定期更新动态数据

## 性能优化建议

1. **索引优化**：已为高频查询字段创建索引
2. **分区表**：如果数据量持续增长，可考虑按时间分区
3. **读写分离**：高并发场景下可配置主从复制
4. **缓存策略**：实时数据可使用Redis缓存，减少数据库查询

## 注意事项

1. 确保MySQL版本 >= 5.7，支持JSON和DATETIME类型
2. 定时任务的cron表达式需要根据实际需求调整
3. API密钥需要妥善保管，不要提交到版本控制系统
4. 生产环境建议使用连接池优化数据库连接
